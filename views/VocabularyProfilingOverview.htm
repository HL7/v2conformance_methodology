Vocabulary Binding and Profiling

Specification of vocabulary conformance is a multidimensional process. It involves successive refinement of both the informational model and vocabulary in parallel. The figure in Section 6 labeled as “Summary of Vocabulary Mechanics” provides an overview of these vocabulary conformance mechanics. The ultimate goal of the specification is to bind coded data elements to a specific set of codes to facilitate a precise implementation.

As described previously, specifications are refined via the process of profiling. The labels in the boxes that are lined up along the left-most side of Figure 1 illustrate a profile hierarchy progressing from a general (abstract) specification to a completely constrained and precise implementation. Each of the three rows, demarcated in the diagram by the horizontal dashed lines, shows the successive operations performed on both the specification of the data elements and the vocabulary objects with which they are associated. This successive constraint is shown in the diagram starting with the top row, then continuing with the next row down, and, finally, ending with the bottom row. To reduce the clutter in the diagram, this process is shown using only a single example data element that is associated with a “simple” value set consisting only of codes from one code system; but all of the indicated items exist for each and every one of the coded data elements in any profile. The binding of a coded data element is either to an internal value set or an external value set (indicated by “XOR” in the diagram).

Figure 1: Vocabulary Binding and Profiling: A High-level Perspective

 

The labels in the three boxes across the top of the diagram indicate the three parallel constraint processes by which specification of vocabulary conformance occurs. The steps related to the process of defining, refining, and maintaining the sets of coded concepts are illustrated in the portion of the diagram that is below the box labelled “Internal Terminology Space”. This information applies to those vocabulary objects that are handled by the same organizational processes used for constraining a profile and/or the base specification. These internal value sets may come from the base specification or may be created as entirely new sets for the profile being developed. The central portion of the diagram below the box labeled "Information Model Space" illustrates the processes wherein the profile definitions are created. These definitions include data elements and their bindings to value sets, as well as various parameters controlling use and extensibility (described in detail in the sections to follow). The portion of the diagram below the box labeled "External Terminology Space" shows that the governance processes for these vocabulary objects are handled by an organizational entity that is outside of, and separate from, the entity developing the profile (although some communication may exist between these entities).

One goal of the profile constraining process is to define a binding to vocabulary that ultimately is implementable for each data element that is assigned a coded element data type. This binding is necessary for all coded data elements for an operational implementation, and the binding can be to a value set that is created and maintained within the confines of the profile definition (internal) or to a value set that is created and maintained externally. An internal binding refers to a value set binding in which the code system is defined within the specification (typically the base standard, but the values may also be defined within any of the profiles being derived from the base standard). This value set is created or copied from another internal source and subsequently bound to a data element. For an external binding, the code system is always maintained by external stewards. Those stewards (or in some cases another third party) may create a value set, or they may copy an external value set expansion into the profile during the final implementation constraining process (as shown within the red dotted line in the diagram).

Examples of external steward organizations include ISO and WHO (which define many international code systems), IHTSDO (SNOMED CT), Regenstrief (LOINC), and VSAC (the Value Set Authority Center of the US National Library of Medicine, which develops and publishes value sets for use in implementing the US “Meaningful Use” regulations). In many (but not all) of these cases, the value sets also are developed externally to the profile constraining activity, but they may have some modifications originating from “inside” the profiling activity. Finally, the labels in the boxes positioned along the rightmost side of the diagram describe the sequential activities that are performed on the code systems and value sets as constraint of the profile moves towards Implementation level refinement.

It is important to note that these three profiling activities shown in parallel may be performed independently and do not have any timing-based relationships between them that can be considered during the iterations of the profiling cycles.

A value set is derived from one or more code systems or another pre-existing value set through the process of constraining and extending (this action is represented in the Figure 1 diagram by the green stars and green arrows). The constraint and extension mechanism applies to both internally and externally defined value sets, and is part of defining and using the binding parameters.

 

The binding of a data element to a value set is annotated in Figure 1 by the blue stars. The specification (i.e., the profile) makes this association explicit. The binding includes parameters that are used to define specificity. This required specificity includes details related to the code system (such as its release version), the requirement to use the codes in the value sets (as opposed to custom codes or free text), and the limits of or guidance on further constraint of the value set for a subsequent profile. These specificity parameters are explained in detail in the sections to follow.

Value set authoring is the initial phase of specification in either of the Terminology Spaces. As a profile is constrained for a specific use case, the value sets are refined to match the needs of that use case. This phase is labeled as “Derived Specification” in the second row of the diagram. The operations on vocabulary during this phase can be described as a “Value Set Modification", where the value sets that were established in the specification to which the constraints are being applied are further constrained (and/or extended); this notion is indicated in Figure 7.3 with the green stars and arrows. Note that extending may involve adding codes from a code system to the value set. The thickness of the binding (black) arrows that point from a data element to a value set box in the diagram decreases in each of the sequential three rows to indicate a more precise (and often smaller) set of values. Note that more than one derived specification may be produced for a set of related or similar use cases during each iteration (i.e., profiling cycle) of the operations shown in the portion of the diagram below the Information Model Space label.

The bottom “Implementation” row represents the implementation profile and its realization (the actual configured system that is in production). In the transition from the “Derived Specification” to the “Implementation”, further constraints (and extensions) may be specified. Additionally, once the value set definition has been finalized, its set of specific values can be realized (e.g., enumerated) through the process of value set expansion. This action, indicated in the diagram by the label "expand()" on the vertical arrow in the second row, is an operation that may be performed manually at the local installation site or by a terminology service that is available to the profile authors, developers,  and system implementers. The value set expansion is the code set instance in a form directly suitable for implementation. Note that under some circumstances, a data element in an Implementation Profile may still be bound to a Value Set Definition that has not yet been expanded to an explicit list of values (“Value Set Expansion”). In these cases, the development of the final Implementation Profile involves further constraint and parameterization of the Value Set Definition, so the final expansion may be completed at the time of implementation of the system, and will be deterministic at that time.
